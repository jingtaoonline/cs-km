Skill summary 的内容是如何设计的？如果 summary 信息不足，模型会不会激活错误的 Skill？你们是怎么评估召回准确率的？

summary 不是对 Skill 功能的完整描述，而是帮助模型做路由决策的触发规则。我们的设计包含三个要素：

  • 明确触发场景：用"当用户说 X 时触发"的句式，列出典型触发词和意图
  • 负样本边界：写清楚"不适用于 Y 场景，应使用 Z Skill"，防止误触发
  • 垂类关键词：餐饮类列出"预订、附近、推荐"；出行类列出"打车、导航、路线"

模型确实可能激活错误的 Skill，有两类保护：对漏触发设置保守阈值，宁可多一次 tool call 按需激活；对误触发在 Skill 全文开头加"适用性检查"，模型加载后可自我判断放弃执行。

Skill 激活准确率具体是怎么评估的？

分两个阶段：

上线前离线评测：人工编写标注 query 集，覆盖"应该触发"和"不应该触发"两类。不应该触发的样本刻意选近似负样本——与 Skill 话题相关但意图不同的 query。比如同样是餐厅场景，"找附近餐厅"应触发 search Skill，"这家餐厅评价怎么样"不该触发。用自动化脚本批量跑，每条 query 跑多次取触发率，处理模型随机性——以触发率是否超过阈值来判断 pass/fail，输出 precision 和 recall 指标，作为上线门槛。

上线后持续审核：线上随机抽样，两层审核：先用更强的模型判断激活是否正确，再人工复核。发现问题后分析是 summary 描述模糊、边界不清，还是模型能力问题，针对性迭代 summary。

按需激活的 tool call 链路上有额外延迟，用户侧能感知到吗？你们是如何在延迟和 Token 消耗之间做权衡的？

用户侧基本没有明显感知，原因是架构上做了两层缓冲。

第一层：入口分流减少了按需激活的频次。简单意图在入口网关就被分流走了，直接走传统搜索+大模型总结，根本不进 Agent 推理链路。只有判断为复杂意图的请求才进入 Agent，这部分用户本来就预期会有思考时间，对延迟的容忍度更高。

第二层：思考过程可视化消化了等待。用户端有思考动画和过程展示，按需激活的那次额外 tool call 发生在"思考"阶段里，用户感知到的是"Agent 在思考"，而不是"系统在卡顿"。

权衡依据是频率和体积的组合判断。高频、体积小的 Skill 直接预加载——反正每次都要用，Token 多花也值；低频、体积大的 Skill 按需激活——大多数请求根本用不到，预加载反而白白消耗 Token。边界目前主要靠人工经验判断，重构期间做了实验对比，验证了这个策略相比全量预加载效果更好。

人工经验判断是否足够准确，有没有更系统的方法？

目前是经验驱动，存在局限。下一步可以数据驱动——统计每个 Skill 的实际激活率和对应的 Token 消耗，用「激活频率 × Skill 体积」来定量决定预加载还是按需激活，动态调整边界。这样既能避免经验偏差，也能随业务变化自动调整策略。

当用户意图跨多个垂类时，会同时激活多个 Skill 吗？这时候上下文管理如何处理？

会同时激活多个 Skill，上下文和执行都有对应的处理机制。

激活层面：系统允许同时激活多个 Skill。跨垂类意图在线上确实发生过，比如用户说"帮我找一家附近的餐厅然后打车过去"，会同时激活餐饮搜索和出行两个 Skill。

上下文层面：多个 Skill 的内容作为 tool call 的返回结果整体加入上下文，没有单独针对 Skill 加载设置 Token 总量上限。兜底依赖更基础的消息裁剪和压缩能力——因为 Skill 内容是作为 tool 结果进入上下文的，和其他消息一样受统一的上下文管理策略约束。所以 Skill 体积控制本身也是一层保护，Skill 全文要尽量精简。

执行层面：执行模块有 DAG 编排，有依赖关系的串行执行，没有依赖的并行执行。比如"找餐厅"和"查打车价格"没有依赖可以并行，"预订餐厅"依赖"找餐厅"的结果则必须串行。

多个 Skill 同时加载，有没有 Token 上限保护？

没有针对 Skill 加载单独设置硬限制。Skill 内容作为 tool 结果进入上下文，和普通消息一样受统一的消息裁剪和压缩策略兜底。间接的保护来自两方面：一是 Skill 体积控制，全文要精简；二是入口分流减少了多 Skill 同时激活的频率。极端场景下的改进方向是设置激活数量上限，或多 Skill 场景下只注入各 Skill 的 summary，按需再展开。

参考 Claude Code 的做法：Claude Code 官方文档明确了多 Skill 共存时的专项预算机制——每个 Skill 最多保留 5,000 tokens，所有 Skill 合计上限 25,000 tokens；上下文压缩时按最近激活优先保留，最早激活的最先被丢弃。这是针对 Skill 的专项 token 管理，比我们目前依赖通用消息压缩更精细。这个方向是我们可以借鉴改进的。

如果参考 Claude Code 的记忆压缩机制，你会如何设计小团的记忆压缩？

参考 Claude Code 的思路，核心借鉴三点：分层压缩顺序、结构化摘要策略、用户可干预机制。但因为小团面向的是普通终端用户而不是开发者，优先级排序和干预方式要做对应调整。

触发机制：自动触发对标 Claude Code，在 context 使用量到达阈值（如 85%）时自动执行。手动触发不暴露技术指令，而是识别用户自然语言——"重新开始"、"换个话题"等——作为主动压缩的信号。

压缩顺序（按可恢复性从低到高裁剪）：

  • 第一层先清：search 结果、API 返回详情等 tool call outputs，可重新获取，优先丢弃
  • 第二层压缩：历史多轮对话，压缩成结构化摘要
  • 第三层保留：system prompt + 用户长期记忆 + 当前会话核心目标

结构化摘要保留内容：

Claude Code 保留小团对应保留
用户意图和目标当前会话用户核心诉求
关键文件路径/代码片段已确认的中间结果（已选餐厅、已定时间等）
报错信息和已尝试的修复已探索方向和用户明确拒绝的选项
待处理任务下一步行动（待用户确认的事项）
重要决策背景用户表达过的偏好（口味、预算、距离等）

用户可干预（对应 Compact Instructions）：Claude Code 让开发者在 CLAUDE.md 里写 ## Compact Instructions 控制保留重点。小团用户不适合暴露这个配置，等价设计是：对话内用自然语言标记优先级——用户说"记住这个很重要"就打高优先级标记，压缩时优先保留；某些业务关键状态（如预订进行到第几步）在 Skill 设计阶段就标记为不可裁剪。

最核心的差异是目标不同：Claude Code 压缩时最怕丢工程上下文导致 AI 写出不一致的代码；小团压缩时最怕破坏用户体验连续性——用户刚说过的偏好被遗忘、刚确认的结果需要重新说。这个优先级差异贯穿了整个设计。

调用大模型进行记忆压缩时，如何处理耗时问题？总结摘要时，如何区分哪些信息应该保留，哪些可以丢弃？

两个问题方向不同，分开回答。

耗时问题：不阻塞主流程，异步执行。

关键是触发时机的选择：压缩不应该在用户发消息时触发，而应该在上一轮回复结束后触发——利用用户阅读和思考回复的空档期执行。有两种具体策略可以组合使用：

  • 空档期预压缩：检测到用户停止输入超过 N 秒时，提前异步触发压缩。大多数情况下用户下次发消息时压缩已经完成，完全无感知。
  • 阈值触发兜底：context 占用到达阈值（如 80%）时必须压缩。此时当前轮回复正常返回，后台异步执行压缩。如果下一条消息在压缩完成前到达，UI 展示"正在整理对话..."状态，轻微延迟但用户有感知预期。

两种策略叠加：平时靠空档期预压缩实现无感知，阈值触发作为兜底保证不超限。

摘要时如何区分保留 vs 丢弃？

核心判断标准：不是"哪些信息重要"，而是"哪些信息不可重建"。

能重新获取的丢掉，丢了就无法复现的必须保留：

信息类型可重建性处理方式
Tool call API 返回详情、搜索结果列表高（可重新调用）优先丢弃
闲聊、寒暄内容高直接丢弃
用户确认过的中间结果低（丢了要重问用户）必须保留
用户明确拒绝的选项低（丢了会重复推荐）必须保留
用户表达的偏好（口味/预算等）低必须保留
当前任务目标低必须保留

但光靠模型自由判断"哪些重要"不够可靠——模型容易把闲聊也总结进去，或漏掉"用户拒绝了 A 方案"这类负样本信息。更好的做法是用强制格式 prompt 让模型填槽，而不是自由摘要：

## 用户当前目标（一句话）
## 已确认的关键信息（逐条列出）
## 用户明确拒绝的选项（防止重复推荐）
## 用户偏好（口味/预算/习惯等）
## 待处理事项（下一步需要确认或执行的）
以上格式之外的内容一律不输出。

强制格式的好处：模型注意力对准关键槽位，输出可以机器解析，方便注入时按优先级取用。

超长对话的质量问题：不要等到快满了才一次性压缩几十轮，改用滚动摘要——每 N 轮做一次增量摘要合并进已有摘要。每次压缩信息量小，摘要质量更稳定，也不会出现单次摘要调用超时的问题。

摘要异步执行时，如果摘要未完成、用户下一条消息就来了导致 token 超限，怎么处理或避免？

根本原因是触发时机设计不当，不是摘要太慢。如果摘要还没完成 context 就被撑爆，说明压缩触发得太晚——解法是提前触发、留足缓冲。

阈值分层设计：80% 触发异步压缩；90% 作为警戒线（此时压缩应已完成，否则说明触发阈值要再提前）；95% 强制等待（压缩未完成则 block 新请求，UI 显示状态）；100% 永远不应该触到。80% 到 100% 之间的余量要能容纳摘要完成前用户可能发来的 2-3 轮对话。

即使阈值设计合理，极端情况（超长消息、连续快速发消息）仍可能来不及，这时需要分级降级兜底：

降级级别做法速度质量
Level 1（最快）直接截断最旧的消息，保留最近 N 条，无需大模型调用毫秒级最差，但不报错
Level 2换轻量模型（如 haiku）做粗粒度摘要，速度提升 5-10 倍秒级较差但可接受
Level 3（正常路径）等待完整摘要完成，UI 显示"正在整理对话..."数秒最好

触发逻辑：context < 95% 正常走异步；95%-98% 换小模型快速摘要；98%-100% 直接截断；到 100% 拒绝请求作为最后保底。

有没有更根本的预防手段，而不只是事后降级？

有，就是滚动摘要——不要等 context 积累到 80% 再一次性处理几十轮，而是每 N 轮对话做一次增量压缩，把这几轮合并进已有摘要。

一次性压缩滚动压缩
单次耗时长，随对话增长劣化短，每次只处理 N 轮
来不及的风险高低
摘要质量随对话长度下降稳定
实现复杂度低中

滚动摘要每次处理的 token 量小，耗时可能只需 1-2 秒，在用户回复间隙就完成了，极大降低"来不及"的概率。这是预防，分级降级是兜底，两者搭配。

长期记忆的提取、存储更新、检索注入流程如何设计？有哪些主流实践？

提取流程分三个阶段：

触发时机：离线异步为主。会话结束后扔进消息队列，异步消费提取，不阻塞主链路。相比在线触发，能看完整会话再提取，质量更高。

提取内容和方式：两类信息分开处理——结构化偏好（过敏原、常用地点、人数等）用槽位提取，存 MySQL；语义性强的自由偏好（"喜欢安静环境"）提炼成自然语言片段，做 embedding 存向量库。两路分开存储，注入时分开检索。

检索注入：会话开始时不全量注入，两路并行检索——意图识别触发对应槽位注入（结构化层），当前 query 向量检索 top-K 相关记忆（向量层）。检索和模型调用并行执行，几乎零额外延迟。控制注入总 token 量上限（如 500 tokens），避免记忆本身撑爆 context。

新记忆和旧记忆冲突怎么处理？

写入前先检索出语义相关的已有记忆，触发一次"记忆合并"调用，输入是新提取记忆 + 已有相关记忆，输出合并结果。合并时需要区分三种情况：

  • 永久更新：用户明确表达偏好变了（"我最近不吃辣了"）→ 覆盖旧记忆
  • 临时覆盖：用户明确说是例外（"今天特殊情况"）→ 本次生效，不写入长期记忆
  • 信号模糊：行为和历史不一致但没说原因 → 降低旧记忆置信度，不立刻覆盖，多次出现才更新

只对"和已有记忆相似度超过阈值"的新记忆触发合并，纯新增直接写入，控制合并调用成本。

参考：Mem0 维护 memory graph，新记忆写入时自动检测冲突节点，触发 merge or update
记忆会不会越积越多？如何老化遗忘？

两种方案，可以结合：

置信度衰减：每条记忆维护 last_confirmed_at 和 confidence 两个字段，confidence 随时间衰减，被用户行为验证时重置到 1.0。检索时按"相似度 × confidence"综合排序，旧记忆自然排序靠后被挤掉。

硬上限淘汰（简化做法）：每用户最多保留 N 条记忆，超出时按 last_confirmed_at 淘汰最旧的。简单够用，生产环境常见。

衰减速度是超参，通过离线实验校准——回放历史会话，看不同衰减速度下记忆命中率变化，选最优值。

向量存储总结文本什么情况下适合用？检索时机是 query 前默认检索还是模型主动 tool call？

向量存储适合：语义性强、模糊表达、用户不会用相同词重复表达的偏好类信息（"喜欢安静环境"、"不喜欢等位"）。不适合：精确事实（地址、手机号）、枚举型属性（过敏原）——这些用结构化存储按字段精确查。

检索时机用默认检索（方案A）为主：对话 Agent 的用户历史偏好几乎每次都有参考价值，默认检索的漏查成本太高。关键是与模型调用并行执行——query 进来同步触发向量检索，等模型第一个 token 出来之前检索结果已经回来了，零额外延迟。

模型主动 tool call（方案B）适合记忆库内容复杂、查询意图不稳定的场景（如知识库/文档库），对话偏好记忆场景不适用。

长期记忆质量如何评估？

评估分三个环节，串联关系，前面坏了后面再好也没用：

环节核心指标测量方法
提取质量精确率、召回率人工标注50-100个对话作 ground truth，对比提取输出
存储质量一致性、新鲜度规则扫描冲突 + 最近行为与记忆吻合率
使用质量命中率、用户满意度A/B 实验（有记忆 vs 无记忆）+ 模型 judge 离线评分

最重要的负向指标：用户主动纠正次数。用户说"不是这个"、"我不喜欢这种"——这是记忆使用错误的直接信号，作为线上常态监控指标。

记忆命中率的测量方式：在模型回复后跑轻量检测，检查回复内容是否和注入的记忆条目语义一致。比如注入了"用户不吃辣"，模型推荐了清淡餐厅为命中，推荐了川菜为未命中。

答题结构：先说分三个环节建立框架 → 每环节说核心指标+测量方法 → 重点讲使用质量 → 落地到"用户纠正次数作为线上监控指标"
注入了记忆但模型没有使用，怎么办？

分两层处理：

短期（提高遵守率）：把关键禁忌（过敏原、硬性限制）单独高亮，不混在普通记忆里。在 system prompt 里用独立区块标注"以下是用户的硬性限制，必须遵守"，和普通偏好记忆区分。

长期（评估驱动迭代）：抽样检查"注入了记忆的会话里，模型有没有遵守"，把违反率作为指标持续监控。发现系统性违反时，分析是记忆注入位置不合理、还是记忆表达方式不够明确，针对性调整。

协议本质与分层
MCP 协议本质是什么？可以简单理解为 HTTP + JSON 格式约定吗？

方向对，但不完整。MCP 更准确的定位是：建立在多种传输层之上的 JSON-RPC 2.0 协议，加上面向 AI Agent 的能力语义约定。

分层结构：

层级内容
应用语义层MCP 定义的 tools / resources / prompts 三种能力抽象
消息格式层JSON-RPC 2.0（固定信封格式：jsonrpc、id、method、params/result/error）
传输层stdio（本地进程）/ HTTP+SSE（远程）/ Streamable HTTP（新规范）

和"普通 REST API"的本质区别：

普通 REST APIMCP
传输HTTPstdio / HTTP / WS
格式自定义 JSONJSON-RPC 2.0
能力发现需要文档/OpenAPI运行时 tools/list 自描述
设计目的人→服务AI Agent→工具

HTTP 只是传输选项之一，核心价值在于让 LLM 能在运行时动态发现并调用工具，不需要提前把工具描述硬编码进系统提示。

MCP Server 的 stdio 和 SSE 两种传输方式，可以认为 stdio 是本地进程通信、SSE 是跨机器网络通信吗？

基本正确，但需要补充：SSE 不只用于跨机器，本地也可以用 SSE；跨机器则只能用 SSE/HTTP。

stdio：MCP client（如 Claude Desktop）直接 spawn 子进程，通过 stdin/stdout 管道交换 JSON-RPC 消息。限制：client 和 server 必须在同一台机器。

SSE：本质是 HTTP 的单向推送。MCP 用它实现非对称双向通道——client→server 用普通 HTTP POST，server→client 用 SSE 长连接。可以跨机器，也可以本地（调试方便）。

场景传输方式
Claude Desktop / Claude Code 调用本地工具stdio
团队共享远程 MCP 服务SSE / HTTP
K8s 部署的 MCP 服务SSE / HTTP
本地调试想用 HTTP 工具抓包SSE（更方便）

结论：跨机器必须 SSE，本地推荐 stdio（更高效）。

MCP Server 能自己指定协议方式吗？比如不用 SSE 流式，改成结果全 ready 后一次返回，或者用 WebSocket 保持长连接？

一次性返回：可以，MCP 2025 新规范已支持。

MCP 2025-03-26 版本新增 Streamable HTTP 传输。Server 可以自己选择：流式响应返回 Content-Type: text/event-stream，一次性响应返回 Content-Type: application/json，client 通过 Content-Type 判断。Spring AI MCP 已跟进此规范。

WebSocket：官方规范不支持，但技术上可自行实现。

MCP 官方传输层只有三种：stdio（标准）、HTTP+SSE（旧，deprecated）、Streamable HTTP（新，推荐）。WebSocket 不在官方规范内。JSON-RPC 2.0 本身传输无关，自己实现 WebSocket transport 完全可行，但：标准 client（Claude Desktop 等）不认；只能用在自己同时控制 client/server 的场景。

场景推荐
工具调用结果小、响应快Streamable HTTP（一次性 JSON）
工具调用耗时长、需进度Streamable HTTP（SSE 流式）
本地部署stdio
自己同时控制 client/server可做 WebSocket，但没必要
JSON-RPC 2.0 是什么协议？是谁发布维护的？在网络协议中属于哪个层级？

是什么：轻量级远程过程调用（RPC）协议，规定用 JSON 格式描述"调用哪个方法、传什么参数、返回什么结果"的信封格式。

谁发布：JSON-RPC Working Group，非正式开源社区组织，不是 IETF/W3C。规范托管在 jsonrpc.org，2010 年发布 2.0 版本，没有 RFC 编号，属于事实标准。十几年基本未变。

协议层级：严格来说不属于 OSI 任何一层，是运行在应用层协议（HTTP/WebSocket/stdio）之上的消息格式规范。

// 请求
{"jsonrpc": "2.0", "id": 1, "method": "tools/call", "params": {...}}

// 成功响应
{"jsonrpc": "2.0", "id": 1, "result": {...}}

// 错误响应
{"jsonrpc": "2.0", "id": 1, "error": {"code": -32601, "message": "Method not found"}}

// 通知（无需响应，无 id）
{"jsonrpc": "2.0", "method": "notifications/progress", "params": {"progress": 50}}

整个规范不到 A4 纸 10 页。MCP、LSP（VS Code 插件协议）、Ethereum JSON-RPC 底层都是它。越简单越容易成为标准。

和普通前后端接口的区别：本质上很相似——区别只是把方法名从 URL path 挪到了 body 里，加了几个固定字段。存在的价值是标准化：统一错误码、id 字段支持异步多路复用、notification 语义（发出去不需要响应）。

SSE 和 HTTP 是什么关系？SSE 是 HTTP 的进一步封装吗？SSE 和 WebSocket 的本质区别是什么？

SSE 和 HTTP 的关系：SSE 就是 HTTP，不是独立协议。

SSE 是基于普通 HTTP 响应的使用约定，只做了两件事：

  • 响应头加一个字段：Content-Type: text/event-stream
  • 响应体不关闭，持续往里写固定格式文本：data: {...}\n\n

没有新协议，没有握手，没有帧格式，完全复用 HTTP。

SSE vs WebSocket 本质区别：

SSEWebSocket
协议独立性始终在 HTTP 内，从未离开借用 HTTP 握手升级，然后脱离 HTTP
通信方向单向（server → client 推送）真正双向
client→server额外的 HTTP POST同一连接直接发
协议复杂度极低，普通 HTTP 即可有独立帧格式协议

WebSocket 是"借用 HTTP 做升级握手，然后脱离 HTTP"；SSE 是"始终在 HTTP 内，从未离开"。

一句话：SSE 不是独立协议，是 HTTP 的一种使用姿势：保持响应不关闭 + 约定文本格式。WebSocket 才是真正脱离 HTTP 的新协议。
MCP 用 SSE 时，client 和 server 通信是纯 SSE 吗？

不是纯 SSE，而是 HTTP POST + SSE 组合：

  • Client → Server：普通 HTTP POST（发请求）
  • Server → Client：SSE 长连接（推响应/通知）

SSE 本身是单向的（只能 server 推 client），MCP 用这个非对称组合实现了逻辑上的双向通道。

阿里 HR 的经典压力测试——被质疑时如何应对？

有时 HR 会故意质疑你来测试反应，例如：

  • 「你这个项目其实也没什么特别的，市面上类似方案很多啊」
  • 「你说 Token 降了 70%，但如果算上额外的 tool call 延迟，其实没省多少吧？」

应对三原则：

  • 不要争辩，不要上头——千万别说「您不懂技术」「我们的方案是有独特性的」
  • 先承认对方说得有道理——「您说得对，确实有这个问题」——展现理性和成熟
  • 再客观补充——「我当时也考虑到了，所以做了 X 来平衡」——展现技术深度

HR 考察的是被质疑时的情绪控制能力和理性回应能力，不是要赢你。

稳定性和动机——HR 在担心什么？

淘宝业务相对传统，HR 会担心：

  • 你是不是只是为了涨薪跳槽？→ 回答中要持续强调「成长诉求」和「业务匹配度」，而非薪酬
  • 你做了 Agent 会不会觉得淘宝业务不够「AI 前沿」？→ 展现对电商场景 Agent 落地复杂度的理解，说明你认为这不是退步
  • 你能不能接受比创业公司更重的流程和协作成本？→ 表达对大团队协作的理解和认可，美团经历本身就是加分项
